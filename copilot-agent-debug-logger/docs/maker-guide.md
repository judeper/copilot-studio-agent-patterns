# Copilot Agent Debug Logger — Maker Guide
> Use this once you've completed `docs/deployment-guide.md`.
> Quick Start: ≈20 minutes for first-time setup; <5 minutes once tools are installed (D9).

This guide is for makers who want a practical debug trail for Copilot Studio agents and the Power Automate flows around them.
It shows the fastest Hello World path first, then the reusable patterns you can add to real agents.

## What you will build
You will write rows to the `cr_agenttrace` Dataverse table from two places:

- Power Automate wrappers around agent calls and tool flows.
- Copilot Studio topic templates that log chain-of-thought notes or conversation history.

The model-driven app, **Agent Debug Console**, reads those rows through three views:

- **Recent Traces** — latest rows, newest first.
- **Timeline by Correlation ID** — `cr_correlationid` → `cr_sequence` → `createdon`.
- **Errors Only** — rows where `cr_status = ERROR`.

## Before you start
Complete these prerequisites once per environment.

| Requirement | Why it matters |
|---|---|
| `docs/deployment-guide.md` completed | Imports the solution, creates the table, and registers flows. |
| `docs/phase-0-mda-authoring.md` completed | Publishes **Agent Debug Console** and the required views. |
| `cr_DebugLoggerEnabled = true` during testing | The flows no-op when this environment variable is false. |
| Permission to run Power Automate child flows | Pattern A and Pattern B write through `flow-1-log-agent-trace`. |
| Permission to edit the consumer agent | Pattern C and Pattern D import topics and may edit instructions. |

Keep this POC framing in mind:

- Payloads are raw.
- Payloads are not redacted.
- Payloads are truncated before Dataverse write.
- The customer adds PII redaction, retention, sampling, and per-user access controls before UAT or production.

## Data contract at a glance
`cr_agenttrace` has 13 custom columns.
The logger also uses Dataverse system columns such as `createdon`, `modifiedon`, and `OwnerId`.

| Column | Purpose |
|---|---|
| `cr_traceid` | Auto-generated row GUID. |
| `cr_tracelabel` | Human label generated by the flow as `source/step @ timestamp`. |
| `cr_correlationid` | Main stitching key across related rows. |
| `cr_agentname` | Friendly name of the calling agent. |
| `cr_source` | Choice: `POWER_AUTOMATE_FLOW`, `TOOL_FLOW`, or `COPILOT_TOPIC`. |
| `cr_sourcename` | Name of the flow or topic that emitted the row. |
| `cr_stepname` | Name of the specific step being logged. |
| `cr_direction` | Choice: `REQUEST`, `RESPONSE`, or `EVENT`. |
| `cr_sequence` | Caller-supplied per-caller tie-breaker. Not a global counter. |
| `cr_payload` | JSON/text payload, raw and truncated to 900,000 characters. |
| `cr_durationms` | Step duration in milliseconds, usually on `RESPONSE` rows. |
| `cr_status` | Choice: `OK` or `ERROR`. |
| `cr_errormessage` | Optional error details when `cr_status = ERROR`. |

## Quick Start (Hello World)
Goal: manually write one row and see it in **Agent Debug Console**.

Expected time:

- ≈20 minutes for first-time setup.
- <5 minutes once tools are installed.

### 1. Confirm prerequisites
1. Open `docs/deployment-guide.md` and complete the import and deploy steps.
2. Open `docs/phase-0-mda-authoring.md` and confirm **Agent Debug Console** exists.
3. Navigate to `https://make.powerapps.com`.
4. Top bar → environment selector → choose the target environment.
5. Left nav → **Solutions**.
6. Open **Copilot Agent Debug Logger**.
7. Select **Environment variables**.
8. Select **cr_DebugLoggerEnabled**.
9. Select **Edit current value**.
10. Set the value to `true`.
11. Select **Save**.
12. Solution command bar → **Publish all customizations**.

Exact env-var click path:

`https://make.powerapps.com → Solutions → Copilot Agent Debug Logger → Environment variables → cr_DebugLoggerEnabled → Edit current value → true`

### 2. Copy the sample input JSON
The current child flow exposes 11 trigger inputs.
Older planning shorthand called this the manual form; use all 11 fields below.

```json
{
  "correlation_id": "test-001",
  "agent_name": "Smoke Test Agent",
  "source": "POWER_AUTOMATE_FLOW",
  "source_name": "manual-smoke-test",
  "step_name": "HelloWorld",
  "direction": "EVENT",
  "sequence": 1,
  "payload": "{\"hello\":\"world\"}",
  "duration_ms": 0,
  "status": "OK",
  "error_message": ""
}
```

Field notes:

| Input | Hello World value | Notes |
|---|---|---|
| `correlation_id` | `test-001` | Used to find the row in the console. |
| `agent_name` | `Smoke Test Agent` | Any friendly test label. |
| `source` | `POWER_AUTOMATE_FLOW` | Matches the Dataverse choice mapping. |
| `source_name` | `manual-smoke-test` | The originating flow or test harness. |
| `step_name` | `HelloWorld` | The inspected step. |
| `direction` | `EVENT` | Manual smoke row is an event. |
| `sequence` | `1` | First row in this one manual caller. |
| `payload` | `{"hello":"world"}` | JSON serialized as a string. |
| `duration_ms` | `0` | No measured duration for the smoke row. |
| `status` | `OK` | Smoke row succeeded. |
| `error_message` | empty string | Only populate on `ERROR`. |

### 3. Run `flow-1-log-agent-trace` manually
1. Navigate to `https://make.powerautomate.com`.
2. Top bar → environment selector → choose the same environment.
3. Left nav → **My flows**.
4. Search for `flow-1-log-agent-trace`.
5. Open **flow-1-log-agent-trace**.
6. Command bar → **Test**.
7. Select **Manually**.
8. Select **Test**.
9. If the test pane offers **Use sample payload**, paste the JSON block above.
10. If the test pane shows individual fields, fill the 11 inputs exactly as shown above.
11. Select **Run flow**.
12. Wait for the run status to complete.
13. Confirm the flow run ended as **Succeeded**.

If the flow run succeeds but no row appears, check `cr_DebugLoggerEnabled` first.
When the variable is false or unreadable, the child flow exits gracefully without writing.

### 4. Open Agent Debug Console
1. Navigate to `https://make.powerapps.com`.
2. Top bar → environment selector → choose the same environment.
3. Left nav → **Apps**.
4. Search for **Agent Debug Console**.
5. Select **Play**.
6. In the app navigation, select **Traces** → **Inspection** → **Agent Trace**.
7. Open the **Recent Traces** view.
8. Confirm one new row appears with `cr_correlationid = test-001`.
9. Open the row.
10. Confirm `cr_payload` contains `{"hello":"world"}`.

Hello World is confirmed when you see exactly one new smoke-test row.

## Common Setup
Use this checklist before adding any pattern to a real agent.

### 1. Import the solution
Follow `docs/deployment-guide.md` Step 4.
The solution name is **Copilot Agent Debug Logger**.
The unique name is `CopilotAgentDebugLogger`.

### 2. Enable the environment variable
1. Navigate to `https://make.powerapps.com`.
2. Top bar → environment selector → choose the target environment.
3. Left nav → **Solutions**.
4. Open **Copilot Agent Debug Logger**.
5. Select **Environment variables**.
6. Select **cr_DebugLoggerEnabled**.
7. Select **Edit current value**.
8. Set the current value to `true`.
9. Select **Save**.
10. Solution command bar → **Publish all customizations**.

Turn it back to `false` when active debugging ends.

### 3. Register the tool flow as an Action on each consumer agent
Pattern C and Pattern D need the tool flow available inside the consumer agent.

1. Navigate to `https://copilotstudio.microsoft.com`.
2. Open the consumer agent.
3. Left nav → **Tools** or **Actions**.
4. Select **+ Add a tool**.
5. Choose **Flow**.
6. Select **tool-log-agent-trace**.
7. Confirm the flow inputs are visible:
   - `correlation_id`
   - `agent_name`
   - `source`
   - `source_name`
   - `step_name`
   - `direction`
   - `sequence`
   - `payload`
   - `duration_ms`
   - `status`
   - `error_message`
8. Save the agent.
9. Publish the agent.

If you imported substituted topic YAML from `dist/topics`, the topic already contains the flow ID.
If you hand-build in the web UI, bind the action to the same `tool-log-agent-trace` flow.

### 4. Open the console before testing
Keep **Agent Debug Console** open in another tab.
Use **Recent Traces** for smoke testing.
Use **Timeline by Correlation ID** when inspecting a multi-row call chain.
Use **Errors Only** when troubleshooting failed actions.

### 5. Payload truncation (A15)
Both `flow-1-log-agent-trace` and `tool-log-agent-trace` truncate `payload` before the Dataverse write.

Implementation pattern:

```text
left(string(...), 900000)
```

Why 900,000 characters:

- `cr_payload` is a Dataverse multiline text column with a max near 1 MB.
- 900,000 leaves headroom for encoding and connector overhead.
- Truncation prevents a large model response from failing the trace write.

Important security note:

- The payload is raw.
- The payload is not redacted.
- Customer-owned PII redaction belongs in UAT or production hardening.
- Use `example.com` domains in samples and fixtures.

## Pattern A — Wrap `ExecuteAgentAndWait` in a Power Automate flow
### Use when
Your Power Automate flow invokes a Copilot Studio agent via the `ExecuteAgentAndWait` action and you want to capture the exact request and response payloads.

### Scope callout
Pattern A covers Power Automate → Agent invocations only.
Agents invoked via Direct Line, Teams, or web chat do not have a Power Automate-side wrapper.
For those channels, use Pattern C for chain-of-thought logging and Pattern D for conversation history capture.

### Shape
Emit two rows around the `ExecuteAgentAndWait` action:

| Row | Direction | Sequence | Payload |
|---|---|---:|---|
| Before the action | `REQUEST` | `1` | Serialized request sent to the agent. |
| After the action | `RESPONSE` | `2` | Serialized response returned by the agent. |

### Click path — REQUEST row before `ExecuteAgentAndWait`
1. Open `https://make.powerautomate.com`.
2. Open your existing calling flow.
3. Locate the `ExecuteAgentAndWait` action.
4. Just before that action, select **+ New step**.
5. Search for `Run a Child Flow`.
6. Select **Run a Child Flow**.
7. Choose **flow-1-log-agent-trace**.
8. Fill the child-flow inputs.

Suggested values:

| Input | Value |
|---|---|
| `correlation_id` | Use Pattern E value if available; otherwise leave blank and let the child flow default. |
| `agent_name` | Your consumer agent display name. |
| `source` | `POWER_AUTOMATE_FLOW` |
| `source_name` | Your calling flow name. |
| `step_name` | `ExecuteAgentAndWait` |
| `direction` | `REQUEST` |
| `sequence` | `1` |
| `payload` | `string(outputs('Compose_AgentInput'))` or your equivalent input compose. |
| `duration_ms` | `0` |
| `status` | `OK` |
| `error_message` | empty string |

Pasteable JSON reference:

```json
{
  "correlation_id": "@{outputs('Compose_CorrelationId')}",
  "agent_name": "Customer Support Agent",
  "source": "POWER_AUTOMATE_FLOW",
  "source_name": "pa-call-support-agent",
  "step_name": "ExecuteAgentAndWait",
  "direction": "REQUEST",
  "sequence": 1,
  "payload": "@{string(outputs('Compose_AgentInput'))}",
  "duration_ms": 0,
  "status": "OK",
  "error_message": ""
}
```

Use `example.com` if your sample payload needs an email address, such as `user@example.com`.

### Click path — RESPONSE row after `ExecuteAgentAndWait`

1. Immediately after `ExecuteAgentAndWait`, select **+ New step**.
2. Search for `Run a Child Flow`.
3. Select **Run a Child Flow**.
4. Choose **flow-1-log-agent-trace**.
5. Fill the child-flow inputs.

Suggested values:

| Input | Value |
|---|---|
| `correlation_id` | Same value as the REQUEST row. |
| `agent_name` | Your consumer agent display name. |
| `source` | `POWER_AUTOMATE_FLOW` |
| `source_name` | Your calling flow name. |
| `step_name` | `ExecuteAgentAndWait` |
| `direction` | `RESPONSE` |
| `sequence` | `2` |
| `payload` | `string(outputs('Execute_an_agent')?['body'])` or your action's body output. |
| `duration_ms` | `div(sub(ticks(utcNow()), ticks(outputs('Compose_StartTime'))), 10000)` |
| `status` | `OK` or `ERROR` |
| `error_message` | Empty for success; action error details for failure. |

Pasteable JSON reference:

```json
{
  "correlation_id": "@{outputs('Compose_CorrelationId')}",
  "agent_name": "Customer Support Agent",
  "source": "POWER_AUTOMATE_FLOW",
  "source_name": "pa-call-support-agent",
  "step_name": "ExecuteAgentAndWait",
  "direction": "RESPONSE",
  "sequence": 2,
  "payload": "@{string(outputs('Execute_an_agent')?['body'])}",
  "duration_ms": "@{div(sub(ticks(utcNow()), ticks(outputs('Compose_StartTime'))), 10000)}",
  "status": "OK",
  "error_message": ""
}
```

### Capturing duration

Add a **Compose** action before the REQUEST row:

```text
utcNow()
```

Name it `Compose_StartTime`.
Then use this expression on the RESPONSE row:

```text
div(sub(ticks(utcNow()), ticks(outputs('Compose_StartTime'))), 10000)
```

### Payload truncation reminder

You do not need to truncate before calling the child flow.
The child flow truncates with `left(string(triggerBody()?['payload']), 900000)`.
Still, avoid logging unnecessary fields because the payload is raw and not redacted.

### ⚠️ `cr_sequence` per-caller-only warning (A8/D4)

The `sequence` integer ties rows within a single caller, such as this Power Automate flow run.
It is not a global step counter across Power Automate and Copilot Studio callers.
Two flows running in parallel can both emit `sequence: 1` for their first row.
Use `cr_correlationid` plus `createdon` to stitch a multi-caller timeline.

### ⚠️ End-user-utterance accidental trigger warning (D4)

Do not populate `correlation_id` from user-typed text.
A user utterance might collide with another conversation's ID and merge unrelated timelines.
Use `System.Conversation.Id` through Pattern E, or a deterministic generated ID controlled by the caller.

### Validation

After running the calling flow:

1. Open **Agent Debug Console**.
2. Select **Timeline by Correlation ID**.
3. Search for the correlation ID.
4. Confirm the REQUEST row appears before the RESPONSE row.
5. Confirm both rows have `cr_source = POWER_AUTOMATE_FLOW`.

## Pattern B — Wrap a tool flow

### Use when

Your custom tool flow is called from a Copilot Studio agent through **Call an action** and you want to inspect that tool's request and response.

Examples:

- A tool flow searches SharePoint.
- A tool flow creates a support ticket.
- A tool flow checks a Dataverse record.
- A tool flow calls an HTTP API.

### Shape

Emit two rows inside the tool flow:

| Row | Where | Direction | Sequence |
|---|---|---|---:|
| Tool entry | Immediately after the trigger | `REQUEST` | `1` |
| Tool exit | Just before **Respond to agent** | `RESPONSE` | `2` |

### Click path — REQUEST row at trigger

1. Open `https://make.powerautomate.com`.
2. Open the tool flow you want to inspect.
3. Locate the trigger **When an agent calls the flow**.
4. Immediately after the trigger, select **+ New step**.
5. Search for `Run a Child Flow`.
6. Select **Run a Child Flow**.
7. Choose **flow-1-log-agent-trace**.
8. Fill the child-flow inputs.

Pasteable JSON reference:

```json
{
  "correlation_id": "@{triggerBody()?['correlation_id']}",
  "agent_name": "Tool Consumer Agent",
  "source": "TOOL_FLOW",
  "source_name": "tool-search-example-records",
  "step_name": "ToolEntry",
  "direction": "REQUEST",
  "sequence": 1,
  "payload": "@{string(triggerBody())}",
  "duration_ms": 0,
  "status": "OK",
  "error_message": ""
}
```

### Click path — RESPONSE row before `Respond to agent`

1. Locate the final **Respond to agent** action.
2. Immediately before it, select **+ New step**.
3. Search for `Run a Child Flow`.
4. Select **Run a Child Flow**.
5. Choose **flow-1-log-agent-trace**.
6. Fill the child-flow inputs.

Pasteable JSON reference:

```json
{
  "correlation_id": "@{triggerBody()?['correlation_id']}",
  "agent_name": "Tool Consumer Agent",
  "source": "TOOL_FLOW",
  "source_name": "tool-search-example-records",
  "step_name": "ToolExit",
  "direction": "RESPONSE",
  "sequence": 2,
  "payload": "@{string(outputs('Compose_ResponseBody'))}",
  "duration_ms": "@{div(sub(ticks(utcNow()), ticks(outputs('Compose_StartTime'))), 10000)}",
  "status": "OK",
  "error_message": ""
}
```

### Error rows

If your tool flow has a failure branch, write a final row with `status = ERROR`.

```json
{
  "correlation_id": "@{triggerBody()?['correlation_id']}",
  "agent_name": "Tool Consumer Agent",
  "source": "TOOL_FLOW",
  "source_name": "tool-search-example-records",
  "step_name": "ToolExit",
  "direction": "RESPONSE",
  "sequence": 2,
  "payload": "@{string(outputs('Compose_ErrorBody'))}",
  "duration_ms": "@{div(sub(ticks(utcNow()), ticks(outputs('Compose_StartTime'))), 10000)}",
  "status": "ERROR",
  "error_message": "@{outputs('Compose_ErrorMessage')}"
}
```

### Payload truncation reminder

The same A15 cap applies here.
The child flow truncates the tool payload to 900,000 characters before writing to `cr_payload`.
Do not rely on this as a security feature; it is a size guard only.

### ⚠️ `cr_sequence` per-caller-only warning (A8/D4)

The `sequence` integer ties rows within this one tool flow run.
It is not a global sequence across the agent, other tools, or other Power Automate flows.
Two tool flows can both emit `sequence: 1` and both be correct.
Use `cr_correlationid` plus `createdon` for cross-caller review.

### ⚠️ End-user-utterance accidental trigger warning (D4)

Do not copy raw user text into `correlation_id`.
If the tool receives a user utterance such as a case number or free-form phrase, keep it inside `payload`, not the correlation key.
Use a caller-supplied generated ID, `System.Conversation.Id`, or the child-flow fallback.

### Validation

After invoking the tool through the agent:

1. Open **Agent Debug Console**.
2. Select **Timeline by Correlation ID**.
3. Search for the conversation or caller-supplied correlation ID.
4. Confirm the tool entry row appears before the tool exit row.
5. Confirm both rows have `cr_source = TOOL_FLOW`.

## Pattern C — Import CoT topic + agent instruction snippet

### Use when

You want Copilot Studio-side capture of the orchestrator's intermediate chain-of-thought-style reasoning after agent steps.
This is the closest match to the Power CAT Custom Engine blog pattern from Oct 2025.

### ⚠️ Cost & Limits callout

- Adds roughly 180 characters to your agent's system prompt.
- Adds one extra tool call per agent turn in the full variant.
- Extra tool calls consume Copilot credits and add latency.
- Requires generative orchestration; classic dispatch agents will not auto-invoke the topic from instructions.
- Keep `cr_DebugLoggerEnabled = false` outside active debugging windows.

### Choose your variant (D6)

| Variant | Behavior | Best for |
|---|---|---|
| `log-chain-of-thoughts.topic.mcs.yml` | Italic chat trace plus Dataverse persistence through `tool-log-agent-trace`. | Forensic debugging with searchable history. |
| `log-chain-of-thoughts-blog-pure.topic.mcs.yml` | Italic chat trace only; zero dependencies. | Live coaching sessions without backend changes. |

The full variant always calls the tool flow.
The tool flow decides whether to write by reading `cr_DebugLoggerEnabled`.
The blog-pure variant never calls Dataverse.

### Click path — import the topic

1. Navigate to `https://copilotstudio.microsoft.com`.
2. Open the consumer agent.
3. Left nav → **Topics**.
4. Select **+ New topic**.
5. Select **From YAML**.
6. Paste the substituted YAML from `dist/topics/log-chain-of-thoughts.topic.mcs.yml` after `docs/deployment-guide.md` Step 7.
7. For the zero-deps path, paste `copilot-studio/topics/log-chain-of-thoughts-blog-pure.topic.mcs.yml` directly.
8. Select **Save**.
9. Select **Publish**.

### YAML snippet

Full variant core binding:

```yaml
- kind: InvokeFlowAction
  id: invokeFlowAction_LogCoT
  flowId: "{{TOOL_LOG_AGENT_TRACE_FLOW_ID}}"
  input:
    binding:
      source: "COPILOT_TOPIC"
      source_name: "log-chain-of-thoughts"
      step_name: "CoT"
      direction: "EVENT"
      payload: =Topic.CoT
      correlation_id: =System.Conversation.Id
```

Blog-pure variant core action:

```yaml
- kind: SendActivity
  id: sendActivity_CoTBlogPure
  activity: "*{Topic.CoT}*"
```

### Click path — add the agent instruction snippet

1. Open the consumer agent in Copilot Studio.
2. Left nav → **Overview** or **Instructions**.
3. Open the agent's **Instructions** editor.
4. Add this snippet at the end.
5. Save the agent.
6. Publish the agent.

Instruction snippet from the Power CAT blog:

```text
After every tool, topic, or step you take (except when you are already calling
/Log Chain of Thoughts or other debug/logging topics), log your intermediate
reasoning by calling /Log Chain of Thoughts.
```

### Cross-pattern correlation warning (D3)

Combining Pattern A and Pattern C without Pattern E produces split timelines.
Pattern A rows may use the Power Automate run fallback.
Pattern C rows default to `System.Conversation.Id`.
Those keys can differ even when the rows describe the same user turn.
Use Pattern E when you need one coherent PA + CS timeline.

### Managed-solution agent caveat (A16)

If your consumer agent is in a managed solution, you cannot edit its instructions directly because the instructions section is locked.
Create an unmanaged overlay solution in the same environment, add the consumer agent as an existing component, edit instructions in the overlay, then publish.

### 🚨 Infinite-loop kill-switch (A14 — 3-step recovery)

If the agent enters a tight loop calling `/Log Chain of Thoughts` repeatedly:

1. Power Apps maker → Solutions → Copilot Agent Debug Logger → Environment variables → cr_DebugLoggerEnabled → Edit current value → false → Save → Publish all customizations
2. Copilot Studio → consumer agent → Topics → Log Chain of Thoughts → toggle Active → off
3. Consumer agent → Instructions → delete the "log your intermediate reasoning" line → Save

After recovery, test with the blog-pure variant first.
Then re-enable the full variant only during an active debugging window.

### Validation

1. Start a test conversation with the consumer agent.
2. Ask a simple question that causes the agent to use a tool or topic.
3. Confirm the chat shows an italic trace message.
4. Open **Agent Debug Console**.
5. Select **Recent Traces**.
6. Confirm a row with `cr_sourcename = log-chain-of-thoughts`.
7. Confirm `cr_correlationid` matches the conversation ID when using the full variant.

## Pattern D — Import ConvHistory topic

### Use when

You want to persist the full conversation transcript at a chosen point.
Common choices are end-of-conversation, escalation, or a manual command.

### Choose your variant (D6)

| Variant | Behavior | Best for |
|---|---|---|
| `save-conversation-history.topic.mcs.yml` | Silent capture plus Dataverse persistence through `tool-log-agent-trace`. | Searchable transcript history and post-mortem analysis. |
| `save-conversation-history-blog-pure.topic.mcs.yml` | Captures the conversation history into a topic variable only. | Verifying the variable populates before wiring downstream actions. |

The full variant writes one `COPILOT_TOPIC` event row.
The blog-pure variant is a safe staging point for makers who want to add their own ticketing, escalation, MCP, or Outlook action later.

### Click path — import the topic

1. Navigate to `https://copilotstudio.microsoft.com`.
2. Open the consumer agent.
3. Left nav → **Topics**.
4. Select **+ New topic**.
5. Select **From YAML**.
6. Paste the substituted YAML from `dist/topics/save-conversation-history.topic.mcs.yml` after `docs/deployment-guide.md` Step 7.
7. For the zero-deps path, paste `copilot-studio/topics/save-conversation-history-blog-pure.topic.mcs.yml` directly.
8. Select **Save**.
9. Select **Publish**.

### YAML snippet

Full variant core binding:

```yaml
- kind: InvokeFlowAction
  id: invokeFlowAction_SaveConvHistory
  flowId: "{{TOOL_LOG_AGENT_TRACE_FLOW_ID}}"
  input:
    binding:
      source: "COPILOT_TOPIC"
      source_name: "save-conversation-history"
      step_name: "ConversationHistory"
      direction: "EVENT"
      payload: =Topic.conversationHistory
      correlation_id: =System.Conversation.Id
```

Blog-pure variant core action:

```yaml
- kind: EndDialog
  id: endDialog_BlogPureComplete
```

### Wiring options

Choose one trigger point.

| Wiring option | Click path | Best for |
|---|---|---|
| Manual | Topic trigger phrase: `save conversation history`. | Maker smoke tests and ad hoc capture. |
| Auto at end | End-of-conversation topic → add a **Recognize Intent** node → choose `/Save Conversation History`. | Capturing every completed support session. |
| On escalation | Escalation topic → before human handoff → add a **Recognize Intent** node → choose `/Save Conversation History`. | Giving a human agent transcript context. |

### 3 downstream extensions

These are not shipped in the POC.
They are maker extension points.

1. **Support ticket creation:** after the `InvokeFlowAction`, add an action to create a row in a ticketing table or a connector such as ServiceNow or Zendesk, with the captured transcript.
2. **Escalation flow:** trigger a separate Power Automate flow that emails or Teams-messages the on-call engineer with the transcript pre-formatted.
3. **MCP / Outlook delivery:** wire to an MCP server that drops the transcript into OneNote, or use the Outlook connector to email the transcript to a shared mailbox such as `support@example.com`.

### Correlation choice

Pattern D uses the same default as Pattern C: `System.Conversation.Id`.
If you also wrap Power Automate calls with Pattern A or tool flows with Pattern B, use Pattern E for one stitched timeline.
Without Pattern E, the console can show separate correlation groups.

### Validation

1. Start a test conversation.
2. Trigger the topic manually by saying `save conversation history`, or reach your configured end/escalation path.
3. Open **Agent Debug Console**.
4. Select **Recent Traces**.
5. Confirm a row with `cr_sourcename = save-conversation-history`.
6. Open the row and confirm `cr_payload` contains the transcript text.

## Pattern E (OPTIONAL) — End-to-end correlation across CS + PA

### Use when

Your debugging scenario requires stitching Power Automate-side rows from Pattern A or Pattern B with Copilot Studio-side rows from Pattern C or Pattern D into one coherent timeline.

### B4 softened claim

`System.Conversation.Id` correlates trace rows within a single conversation only.
It does not guarantee stitching across:

- Parallel conversations, such as two users hitting the same agent at the same time.
- Retries, where a retried turn can produce a new context.
- Multi-agent handoffs, where each agent may own a separate conversation.
- Power Automate-only callers that never receive the Copilot Studio conversation ID.

### For perfect stitching

Pack a `correlation_id` field into the existing serialized JSON input you already send to the agent through `ExecuteAgentAndWait`.
The Intelligent Work Layer precedent is the structured `BRIEFING_INPUT` shape used by its agent prompts.
That pattern keeps the correlation key inside the same payload that already flows through the agent boundary.

Example serialized input:

```json
{
  "correlation_id": "debug-2026-05-19-001",
  "request_type": "briefing",
  "user": {
    "display_name": "Example Maker",
    "email": "maker@example.com"
  },
  "payload": {
    "subject": "Prepare a daily briefing",
    "priority": "normal"
  }
}
```

### Fallback chain

The child flow and topics resolve `correlation_id` in this order:

1. Caller-supplied value — highest priority.
2. `System.Conversation.Id` — topic default.
3. `workflow().run.name` — child-flow default when no caller value exists.

### Power Automate setup

1. In the calling flow, add a **Compose** action named `Compose_CorrelationId`.
2. Use a deterministic ID for the whole scenario.
3. Add that value into the JSON payload sent to `ExecuteAgentAndWait`.
4. Pass the same value into the Pattern A child-flow rows.
5. If a tool flow receives the ID, pass it into Pattern B rows.

Example compose value:

```text
concat('debug-', workflow()?['run']?['name'])
```

Example agent input envelope:

```json
{
  "correlation_id": "@{outputs('Compose_CorrelationId')}",
  "user_email": "maker@example.com",
  "request": {
    "kind": "debug-smoke-test",
    "message": "Hello from Pattern E"
  }
}
```

### Copilot Studio setup

1. Keep Pattern C and Pattern D topic defaults at `=System.Conversation.Id` unless you have a supported way to pass the packed ID into the topic.
2. When your agent calls a tool flow, include `correlation_id` as a tool input if the tool exposes it.
3. Do not ask the end user to type the correlation ID.
4. Keep the ID generated by the caller or by the system.

### Console review

Use **Timeline by Correlation ID**.
That view sorts by:

1. `cr_correlationid`
2. `cr_sequence`
3. `createdon`

This is the right-most stitching shape for the POC.
If two rows have the same `cr_sequence`, read `createdon` as the tie-breaker.

## Pattern selection guide

| Scenario | Use Pattern |
|---|---|
| A Power Automate flow calls `ExecuteAgentAndWait`. | Pattern A |
| A Copilot Studio agent calls a custom tool flow. | Pattern B |
| You need visible intermediate reasoning after each agent step. | Pattern C |
| You need a transcript at the end or on escalation. | Pattern D |
| You need one stitched PA + CS timeline. | Pattern E |

Use only the patterns you need.
For a short live demo, Quick Start plus Pattern C blog-pure is often enough.
For forensic debugging, use Pattern A, Pattern C full, and Pattern E.

## Troubleshooting

### I ran the flow and no row appeared

Check these in order:

1. **Solutions** → **Copilot Agent Debug Logger** → **Environment variables** → `cr_DebugLoggerEnabled` is `true`.
2. The flow run actually reached the `Create_AgentTrace_Row` action.
3. The model-driven app view is not filtered away from your row.
4. Search by `cr_correlationid` in **Timeline by Correlation ID**.

### The topic runs but `logged = false`

That usually means logging is disabled or the tool flow fail-open path returned safely.
Check `cr_DebugLoggerEnabled` first.
Then open the tool flow run history.

### The timeline is split into two groups

You likely combined Power Automate-side logging with Copilot Studio-side logging without passing the same `correlation_id`.
Use Pattern E.

### The payload is cut off

That is expected when the payload exceeds 900,000 characters.
The logger protects the Dataverse multiline field limit.
Log smaller, focused payloads when you need full detail.

### The agent starts logging repeatedly

Use the A14 kill switch immediately:

1. Power Apps maker → Solutions → Copilot Agent Debug Logger → Environment variables → cr_DebugLoggerEnabled → Edit current value → false → Save → Publish all customizations
2. Copilot Studio → consumer agent → Topics → Log Chain of Thoughts → toggle Active → off
3. Consumer agent → Instructions → delete the "log your intermediate reasoning" line → Save

### I cannot edit the agent instructions

If the agent is from a managed solution, use an unmanaged overlay solution.
Add the agent as an existing component, edit instructions there, and publish.

## Security and PII checklist

Before sharing traces outside your dev environment:

- Confirm all sample emails use `example.com`.
- Do not paste real tenant IDs, org URLs, payload logs, or customer domains into docs or fixtures.
- Do not treat truncation as redaction.
- Do not leave `cr_DebugLoggerEnabled = true` after active debugging.
- Decide retention before UAT or production.
- Decide whether per-user or per-agent role scoping is needed.
