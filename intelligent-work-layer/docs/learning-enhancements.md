# Learning Enhancements ‚ÄĒ Design Document

This document describes the learning subsystem enhancements for the Intelligent Work Layer. These features enable the agent to adapt to each user's communication patterns over time ‚ÄĒ bootstrapping from historical data, analyzing edit behavior, applying exponentially weighted averages, and gating learned preferences behind confidence thresholds.

> **Status**: Design phase. These enhancements build on the Sprint 4 sender-adaptive foundation (`cr_senderprofile`, `cr_episodicmemory`, `cr_semanticknowledge`) documented in [agent-flows.md](agent-flows.md).

---

## Graph-Powered Silent Bootstrap

### Purpose

Eliminate the cold-start problem by importing 90 days of Outlook and Teams history on first run. Without bootstrapping, the system has zero sender profiles and cannot make sender-adaptive triage decisions until sufficient interactions accumulate organically ‚ÄĒ which may take weeks.

### Trigger

First-run detection: the bootstrap flow fires when a query against `cr_senderprofile` for the authenticated user returns zero rows.

### Flow Steps

**1. Query Outlook Messages (Graph API)**

```
GET /me/messages?$filter=sentDateTime ge {90_DAYS_AGO}&$top=500&$select=sender,toRecipients,ccRecipients,subject,sentDateTime,receivedDateTime,conversationId&$orderby=sentDateTime desc
```

> **Note**: Only metadata fields are requested (`$select`). Message body content is never retrieved. Pagination via `@odata.nextLink` if the mailbox exceeds 500 messages.

**2. Query Teams Messages (Graph API)**

```
GET /me/chats/getAllMessages?$filter=createdDateTime ge {90_DAYS_AGO}&$top=500&$select=from,createdDateTime,chatId,channelIdentity
```

> **Note**: Requires `Chat.Read` delegated permission. Only sender identity and timestamps are extracted ‚ÄĒ message content is not read or stored.

**3. Extract Sender Analytics**

For each unique sender across both channels, compute:

| Metric | Calculation |
|--------|-------------|
| `signal_count` | Total messages received from this sender |
| `response_rate` | (Messages the user replied to) / (Total messages from sender) |
| `avg_response_hours` | Mean time between receiving a message and the user's reply |
| `preferred_channel` | Channel with the most interactions (EMAIL or TEAMS) |

**4. Pre-populate `cr_senderprofile`**

Create one row per unique sender with:

| Field | Value |
|-------|-------|
| `cr_senderemail` | Sender's email address |
| `cr_senderdisplayname` | Sender's display name |
| `cr_sendercategory` | `AUTO_INFERRED` |
| `cr_signalcount` | Computed signal_count |
| `cr_responserate` | Computed response_rate |
| `cr_avgresponsehours` | Computed avg_response_hours |
| `Owner` | Authenticated user's Azure AD Object ID |

**5. Detect VIPs**

Senders matching **both** criteria are upgraded:

- `response_rate` > 0.80
- `avg_response_hours` < 30 minutes (0.5 hours)

These senders are set to `cr_sendercategory = AUTO_HIGH`.

### Privacy

- Only metadata is analyzed: sender address, display name, timestamps, subject lines, and conversation IDs.
- Message body content is **never** retrieved, stored, or processed.
- All data is scoped to the authenticated user's mailbox via delegated permissions.

### Performance

- Duration: ~2‚Äď5 minutes for a typical enterprise mailbox.
- Runs as a background flow (no UI blocking).
- Pagination handles mailboxes with >500 messages gracefully.

## Edit Analyzer Agent

### Purpose

Compare original agent-generated drafts against user-edited versions to classify change patterns. These classifications feed the learning loop ‚ÄĒ the system learns *how* a user modifies drafts for specific senders and adjusts future drafts accordingly.

### Prompt File

[`prompts/edit-analyzer-agent-prompt.md`](../prompts/edit-analyzer-agent-prompt.md)

### Runtime Inputs

| Variable | Type | Description |
|----------|------|-------------|
| `{{ORIGINAL_DRAFT}}` | String | The draft generated by the Draft Generator agent |
| `{{EDITED_DRAFT}}` | String | The version the user actually sent (captured by Flow 5) |
| `{{SENDER_PROFILE}}` | JSON | `cr_senderprofile` row for the recipient |
| `{{TRIGGER_TYPE}}` | String | `EMAIL \| TEAMS_MESSAGE \| CALENDAR_SCAN` |

### Output Schema

```json
{
  "changes": [
    {
      "type": "<TONE_CHANGE | CONTENT_ADDED | CONTENT_REMOVED | SIGNOFF_CHANGED | GREETING_CHANGED | FORMALITY_SHIFT | STRUCTURE_CHANGE>",
      "description": "Human-readable description of the change",
      "before": "Original text segment",
      "after": "Edited text segment"
    }
  ],
  "edit_distance": 0.0,
  "overall_pattern": "Brief summary of the user's editing tendencies for this sender"
}
```

**Field rules:**

- `changes`: Array of classified change objects. May be empty if drafts are identical.
- `edit_distance`: Normalized 0.0‚Äď1.0 score. 0.0 = identical, 1.0 = completely rewritten.
- `overall_pattern`: One-sentence summary (e.g., "User consistently adds personal greetings and shifts to informal tone for this sender").

### Integration

Flow 5 (Outcome Tracking) invokes the Edit Analyzer when the tracked outcome is `SENT_EDITED`. The analysis results are stored in `cr_episodicmemory` as a structured JSON payload, linked to the original card and sender profile.

```
Scope: Analyze Edit
  ‚Ēú‚ĒÄ‚ĒÄ Compose ‚ÄĒ ORIGINAL_DRAFT (from card's draft_payload)
  ‚Ēú‚ĒÄ‚ĒÄ Compose ‚ÄĒ EDITED_DRAFT (from sent items match)
  ‚Ēú‚ĒÄ‚ĒÄ Invoke Edit Analyzer agent
  ‚Ēú‚ĒÄ‚ĒÄ Parse JSON ‚ÄĒ edit analysis result
  ‚ĒĒ‚ĒÄ‚ĒÄ Update cr_episodicmemory row with analysis
```

## EWMA Running Averages

### Formula

All running averages in `cr_senderprofile` use Exponentially Weighted Moving Average (EWMA):

```
new_avg = (őĪ √ó new_value) + ((1 - őĪ) √ó old_avg)
```

Where **őĪ = 0.3** (smoothing factor).

### Rationale

- Recent behavior is weighted at **30%**, historical at **70%**.
- Balances stability (avoids overreacting to outliers) with adaptation (reflects genuine behavior changes within ~5‚Äď10 interactions).
- őĪ = 0.3 was chosen as a middle ground: lower values (0.1) adapt too slowly for role changes; higher values (0.5) overreact to one-off interactions.

### Applied Fields

| Field | Description |
|-------|-------------|
| `cr_avgresponsehours` | Average time to respond to this sender |
| `cr_avgeditdistance` | Average edit distance on drafts for this sender |
| `cr_acceptancerate` | Rate at which drafts are sent as-is or with minor edits |

### Implementation

Power Automate expression in Flow 5 (Outcome Tracking) and Flow 9 (Sender Analytics) Compose steps:

```
@{add(
    mul(0.3, triggerOutputs()?['new_value']),
    mul(0.7, triggerOutputs()?['old_avg'])
)}
```

### Example

If `old_avg_response_time` = 120 min and the user responds in 30 min:

```
new_avg = (0.3 √ó 30) + (0.7 √ó 120)
        = 9 + 84
        = 93 min
```

The average drops from 120 ‚Üí 93 minutes, reflecting the faster response without discarding historical behavior.

## Role Change Reset

### Trigger

User-initiated via the Settings card (Flow 7) or an admin command. Not triggered automatically.

### Purpose

When a user changes roles or departments, learned preferences may no longer apply. This feature resets inferred knowledge while preserving explicit user overrides.

### Steps

**1. Deactivate Inferred Semantic Facts**

```
Filter: cr_source = "INFERRED" AND cr_ownerid = {authenticated_user}
Update: cr_isactive = false
```

All facts with `cr_source = "INFERRED"` are soft-deleted (deactivated, not permanently removed).

**2. Reset Sender Categories**

```
Filter: cr_sendercategory IN ("AUTO_INFERRED", "AUTO_HIGH", "AUTO_LOW", "AUTO_UNKNOWN")
        AND cr_ownerid = {authenticated_user}
Update: cr_sendercategory = "AUTO_UNKNOWN"
```

Senders with `USER_EXPLICIT` or `USER_OVERRIDE` categories are **not** modified.

**3. Preserve User-Explicit Facts**

Facts with `cr_source = "USER_EXPLICIT"` or `cr_source = "USER_OVERRIDE"` remain active and unchanged. These represent deliberate user decisions that should survive a role change.

**4. Trigger Silent Bootstrap**

Re-run the Graph-Powered Silent Bootstrap (see above) to reimport sender analytics with fresh context. New `AUTO_INFERRED` profiles are created alongside preserved explicit ones.

**5. Confirmation Card**

Create a LIGHT-tier card with a reset summary:

```json
{
  "item_summary": "Learning reset complete. Deactivated {N} inferred facts, reset {M} sender categories. Your {K} explicit preferences were preserved.",
  "triage_tier": "LIGHT",
  "priority": "Low",
  "card_status": "READY"
}
```

### Safety

- Requires **double confirmation** via the Settings card: "Are you sure? This resets all learned preferences. Your explicit overrides will be preserved."
- The first confirmation presents the scope (number of facts/senders affected); the second confirms the action.

## Tone Inference Gating

### Rule

Disable learned tone preferences until the draft acceptance rate exceeds 50% for a given sender. This prevents the "uncanny valley" effect where partially-learned tone feels more wrong than a safe default.

### Implementation

In the Triage Agent and Draft Generator prompts, the following logic gates tone inference:

**1. Check Acceptance Rate**

Read `cr_senderprofile.cr_acceptancerate` and `cr_senderprofile.cr_signalcount` for the target sender.

**2. Apply Gate**

| Condition | Tone Used |
|-----------|-----------|
| `acceptance_rate < 0.5` OR `total_interactions < 5` | Default: `SEMI_FORMAL` |
| `acceptance_rate ‚Č• 0.5` AND `total_interactions ‚Č• 5` | Inferred tone from `cr_semanticknowledge` |

**3. Power Automate Expression**

```
@{if(
    and(
        greaterOrEquals(float(outputs('Get_Sender_Profile')?['body/cr_acceptancerate']), 0.5),
        greaterOrEquals(int(outputs('Get_Sender_Profile')?['body/cr_signalcount']), 5)
    ),
    outputs('Get_Semantic_Tone')?['body/cr_factstatement'],
    'SEMI_FORMAL'
)}
```

### Rationale

- Below 5 interactions, the sample size is too small for reliable inference.
- Below 50% acceptance, the agent is getting it wrong more often than right ‚ÄĒ defaulting to a neutral tone is safer.
- Once the threshold is met, the system has enough signal to confidently apply learned preferences.

## Memory Poisoning Cap

### Rule

External senders (outside the user's organization domain) are capped at `sender_category = AUTO_MEDIUM` in Triage. This prevents external actors from gaming the system to consistently receive FULL-tier treatment.

### Implementation

**1. Domain Check**

The Triage Agent checks the sender's email domain against the authenticated user's organization domain:

```
@{if(
    endsWith(
        toLower(outputs('SENDER_EMAIL')),
        concat('@', outputs('Get_Org_Domain')?['body/domain'])
    ),
    'INTERNAL',
    'EXTERNAL'
)}
```

**2. Triage Cap for External Senders**

| External Sender Condition | Max Triage Tier |
|---------------------------|-----------------|
| Corroborated by org signals (shared calendar event, Teams channel membership, existing project thread) | `FULL` |
| No corroboration | `LIGHT` (regardless of other signals) |

**3. Corroboration Signals**

An external sender is corroborated if **any** of the following are true:

- The sender appears on a calendar event with the user in the next 14 days
- The sender is a member of a Teams channel the user belongs to
- The sender has an active conversation thread with ‚Č• 3 prior exchanges in `cr_episodicmemory`

### Rationale

- Prevents external senders from inflating their importance by sending high-frequency or urgent-sounding messages.
- Legitimate external contacts (clients, partners) will naturally have corroborating signals.
- Internal senders are not subject to this cap ‚ÄĒ org membership is itself a trust signal.

## Active Learning Patterns

### Draft Edit Detection

If the user makes **3 or more similar edits** to drafts for the same sender within a 14-day window, the system suggests creating a semantic fact rule.

- **Detection**: Flow 5 (Outcome Tracking) aggregates `cr_episodicmemory` entries where `edit_analysis.changes[].type` matches the same pattern for the same sender within the rolling window.
- **Suggestion**: A new `cr_semanticknowledge` row is created with `cr_source = "SUGGESTED"` and `cr_isactive = false`.
- **Example**: If the user adds a personal greeting 3 times for Alice Chen, the system suggests: *"Always include a personal greeting when drafting for Alice Chen."*

### Auto-SKIP Suggestion

If the user dismisses **5 or more cards** from the same sender within a 14-day window, the system suggests an auto-SKIP rule.

- **Detection**: Flow 5 counts `cr_episodicmemory` entries where `cr_outcome = "DISMISSED"` for the same sender.
- **Suggestion**: A LIGHT-tier card is created: *"You've dismissed 5 cards from {sender} recently. Would you like to auto-skip messages from this sender?"*

### Presentation

- Suggestions are surfaced as **LIGHT-tier cards** with a `"Learning Suggestion"` label in the `item_summary`.
- Cards include Accept / Reject actions.
- Suggestions expire after 7 days if not acted upon.

### Confirmation

- Users must **explicitly accept or reject** suggestions before they become active facts.
- Accepted suggestions are promoted: `cr_source` changes from `"SUGGESTED"` to `"USER_EXPLICIT"`, `cr_isactive` changes to `true`.
- Rejected suggestions are marked `cr_isactive = false` and the same suggestion is suppressed for 30 days.

## Learning Metrics

Design metrics for monitoring learning system health over time. These metrics help determine whether the learning system is genuinely improving the user experience.

### Core Metrics

| Metric | Formula | Target Trend |
|--------|---------|--------------|
| **Acceptance Rate** | Drafts sent as-is or with minor edits (`edit_distance < 0.2`) / Total drafts sent | ‚ÜĎ Increasing |
| **Override Rate** | Triage decisions manually changed by user / Total triage decisions | ‚Üď Decreasing |
| **Edit Distance Trend** | EWMA of `edit_distance` values over trailing 30 days | ‚Üď Decreasing |
| **Time-to-Action** | Average minutes from card `created_on` to user action (`SENT`, `DISMISSED`, etc.) | ‚Üď Decreasing |
| **Memory Utilization** | Active semantic facts (`cr_isactive = true`) / Total semantic facts | Stable 60‚Äď80% |

### Composite: Learning Velocity Score

A single 0.0‚Äď1.0 score summarizing overall learning health:

```
Learning Velocity = (0.3 √ó acceptance_trend)
                  + (0.3 √ó override_decrease)
                  + (0.2 √ó edit_distance_decrease)
                  + (0.2 √ó time_to_action_decrease)
```

Where each component is normalized to 0.0‚Äď1.0:

- `acceptance_trend`: Current acceptance rate minus 30-day-ago acceptance rate (clamped to [0, 1])
- `override_decrease`: 1.0 minus current override rate (higher = fewer overrides = better)
- `edit_distance_decrease`: 1.0 minus current EWMA edit distance (lower distance = better)
- `time_to_action_decrease`: 1.0 minus (current avg time-to-action / baseline time-to-action), clamped to [0, 1]

> **Interpretation**: A score above 0.6 indicates healthy learning. Below 0.4 suggests the learning system may need parameter tuning (e.g., adjusting őĪ, resetting stale profiles).

## Context-Dependent Knowledge (`cr_contextscope`)

### Schema Change

New column on `cr_semanticknowledge`:

| Column | Type | Values |
|--------|------|--------|
| `cr_contextscope` | Choice | `UNIVERSAL`, `INTERNAL_ONLY`, `EXTERNAL_ONLY`, `CLIENT_FACING`, `TEAM_SPECIFIC` |

### Scope Definitions

| Scope | Description | Example Fact |
|-------|-------------|--------------|
| `UNIVERSAL` | Applies to all communications | "Always sign off with full name and title" |
| `INTERNAL_ONLY` | Applies only to internal (same-domain) senders | "Use first names in greetings" |
| `EXTERNAL_ONLY` | Applies only to external senders | "Include company disclaimer in footer" |
| `CLIENT_FACING` | Applies to client communications specifically | "Use formal tone, avoid jargon" |
| `TEAM_SPECIFIC` | Applies within the user's immediate team | "Use emoji in casual updates" |

### Memory Injection

When the Draft Generator or Triage Agent requests semantic knowledge, the memory injection layer filters facts by matching `cr_contextscope` to the current signal context:

1. Determine signal context from sender domain, Teams channel type, and calendar event attendees.
2. Include facts where `cr_contextscope = "UNIVERSAL"` (always included).
3. Include facts where `cr_contextscope` matches the determined context.
4. Exclude facts where `cr_contextscope` conflicts with the determined context (e.g., `INTERNAL_ONLY` facts are excluded for external senders).

## Cross-Channel Preferences (`cr_preferredchannel`)

### Schema Change

New column on `cr_senderprofile`:

| Column | Type | Values |
|--------|------|--------|
| `cr_preferredchannel` | Choice | `EMAIL`, `TEAMS`, `EITHER`, `UNKNOWN` |

### Learning Logic

The preferred channel is inferred from response patterns in `cr_episodicmemory`:

1. For each sender, count responses by channel over the trailing 30 days.
2. If ‚Č• 70% of responses are via one channel, set `cr_preferredchannel` to that channel.
3. If the split is closer to 50/50, set `cr_preferredchannel = "EITHER"`.
4. If insufficient data (< 3 interactions), set `cr_preferredchannel = "UNKNOWN"`.

### Usage

The Draft Generator uses `cr_preferredchannel` to add channel suggestions to draft cards:

- If the current signal is EMAIL but `cr_preferredchannel = "TEAMS"`: append a note ‚ÄĒ *"Consider reaching out via Teams instead ‚ÄĒ {sender} typically responds faster there."*
- If the current signal matches `cr_preferredchannel`: no additional note.
- If `cr_preferredchannel = "UNKNOWN"` or `"EITHER"`: no additional note.

> **Note**: Channel suggestions are advisory only. The system never auto-switches channels ‚ÄĒ the user always decides where to send.

## Graduated Autonomy System

> **Cross-reference**: [Gap 8: No Graduated Autonomy](../.planning/design-review-productivity-noise.md) ¬∑ Priority: P2 ¬∑ Complexity: High

### Problem

The system is either fully manual (user reviews every draft) or nothing. There's no progression from "show me everything" ‚Üí "handle routine stuff yourself." Trust is binary.

### Design

Implement three autonomy tiers stored on `cr_userpersona`:

| Tier | Entry Criteria | Behavior |
|------|---------------|----------|
| **OBSERVER** | Default (first 30 days) | All items shown, all drafts require review. Full transparency. |
| **ASSIST** | `cr_acceptancerate > 0.70` AND `cr_totalinteractions > 50` | Auto-dismiss LIGHT items matching learned SKIP patterns. Auto-send drafts with confidence ‚Č• 95 to AUTO_HIGH senders (with 30-second undo window). |
| **PARTNER** | `cr_acceptancerate > 0.85` AND `cr_totalinteractions > 200` | Auto-send all ‚Č• 90 confidence drafts with 30-second undo window. Surface only exceptions (low confidence, new senders, escalations). |

### Schema

New columns on `cr_userpersona`:

| Column | Type | Description |
|--------|------|-------------|
| `cr_autonomytier` | Choice (OBSERVER / ASSIST / PARTNER) | Current tier. Default: OBSERVER. |
| `cr_totalinteractions` | WholeNumber | Cumulative card interactions. Incremented by Flow 5. |
| `cr_acceptancerate` | Decimal (0‚Äď1) | (SENT_AS_IS + SENT_EDITED) / total outcomes. Recomputed by Flow 5. |

### Tier Transition Flow

Transitions are **suggested by the system, confirmed by the user**:

```
Flow 5 (Card Outcome Tracker) ‚ÄĒ on each outcome:
1. Increment cr_totalinteractions
2. Recompute cr_acceptancerate
3. Check tier eligibility:
   - If OBSERVER and rate > 0.70 and interactions > 50:
     ‚Üí Create a card with trigger_type = "AUTONOMY_SUGGESTION"
       "Your acceptance rate is 74% across 52 interactions. 
        Would you like to enable Assist mode?"
   - If ASSIST and rate > 0.85 and interactions > 200:
     ‚Üí Similar suggestion card for Partner tier
4. User confirms via card action ‚Üí Flow updates cr_autonomytier
```

### Safety Rails

- **Downgrade**: If acceptance rate drops below tier threshold for 2 consecutive weeks, suggest downgrade
- **Override**: User can manually set tier at any time via Sender Management UI
- **Audit**: All auto-actions are logged with `cr_cardoutcome = AUTO_SENT` or `AUTO_DISMISSED` for review
- **Undo window**: All auto-sent drafts have a 30-second undo window (leverages Gap 13 undo toast infrastructure)

### Agent Integration

The Triage Agent receives `{{AUTONOMY_TIER}}` as a runtime input:

- **OBSERVER**: No behavior change ‚ÄĒ standard triage
- **ASSIST**: Agent adds `"auto_action_eligible": true` to output when conditions are met for auto-dismiss or auto-send
- **PARTNER**: Agent adds `"auto_action_eligible": true` with lower confidence threshold

## Cold Start Tone Bootstrap

> **Cross-reference**: [Gap 10: Cold Start Tone Bootstrap](../.planning/design-review-productivity-noise.md) ¬∑ Priority: P2 ¬∑ Complexity: Medium

### Problem

The 5-interaction minimum for tone gating means early drafts feel generic and impersonal. Users dismiss them ‚Üí low acceptance rate ‚Üí system thinks it's wrong ‚Üí vicious cycle.

### Design

Extend the Graph-Powered Silent Bootstrap (described above) to analyze the user's own sent email patterns and seed the `cr_tonebaseline` field on `cr_userpersona`.

### Bootstrap Extension

During the existing Graph bootstrap flow (which fires when `cr_senderprofile` has zero rows), add a Sent Items analysis phase:

```
1. Query Sent Items (Graph API):
   GET /me/mailFolders/sentitems/messages
     ?$filter=sentDateTime ge {90_DAYS_AGO}
     &$top=200
     &$select=body,subject,toRecipients,sentDateTime
     &$orderby=sentDateTime desc

2. For each sent email, extract:
   - Greeting pattern (e.g., "Hi {name},", "Dear {name},", no greeting)
   - Sign-off pattern (e.g., "Best regards,", "Thanks,", "Cheers,")
   - Formality indicators:
     - Contraction usage rate (don't vs. do not)
     - Average sentence length
     - Emoji/exclamation usage rate
   - Response structure (prose vs. bullet points)

3. Compute baseline:
   {
     "formality_score": 0.72,          // 0 = very casual, 1 = very formal
     "greeting_patterns": ["Hi {name},", "Hey {name},"],
     "signoff_patterns": ["Best,", "Thanks,"],
     "avg_sentence_length": 14.2,
     "emoji_usage_rate": 0.03,
     "contraction_rate": 0.68,
     "bullet_preference": 0.35         // % of emails using bullet points
   }

4. Write to cr_userpersona.cr_tonebaseline (JSON blob)
```

### Humanizer Integration

The Humanizer Agent receives `{{TONE_BASELINE}}` as an additional runtime input:

- When `cr_tonebaseline` is populated AND per-sender tone data has fewer than 5 interactions:
  - Use baseline formality, greeting, and sign-off patterns
  - Apply baseline sentence length and structure preferences
- When per-sender data reaches 5+ interactions:
  - Per-sender tone overrides baseline (existing behavior)
  - Baseline remains as fallback for new senders

### Privacy

- Only metadata patterns are extracted (greeting/sign-off/formality statistics). Full email body content is processed in-memory and never stored.
- The `cr_tonebaseline` JSON blob contains statistical aggregates, not email content.
- Users can clear their tone baseline at any time via the Sender Management UI.

### Expected Impact

- Early drafts match the user's natural communication style from day one
- Reduces initial dismissal rate by an estimated 20-30%
- Breaks the cold-start vicious cycle: better drafts ‚Üí higher acceptance ‚Üí faster sender learning
