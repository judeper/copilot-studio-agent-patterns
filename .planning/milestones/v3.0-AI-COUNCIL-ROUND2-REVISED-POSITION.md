# AI Council — Round 2 Revised Position

**Convened by:** AI Council Moderator
**Date:** 2026-03-08
**Context:** Owner push-back on Round 1 rejections of Heartbeat (3.2) and Skill System (3.3). Re-evaluation of all 6 proposed v3.0 enhancements through the "single pane of glass" lens.
**Council:** Enterprise Architect, Product Strategist, Security Lead, Power Platform Engineer

---

## Preamble

The owner's feedback forces us to confront a framing error in Round 1. We evaluated the Heartbeat and Skill System as autonomous-action features and applied risk calculus appropriate for systems that *do things*. The owner is correct: the EWA *prepares things*. It creates cards, not consequences. Every output is a recommendation that sits in the dashboard until a human acts on it. This distinction materially changes the risk profile for both rejected items. We also under-weighted the core value proposition — the single pane of glass — which reframes "nice-to-have" features as structurally necessary for completeness. This revised position corrects both errors while preserving the council's valid concerns about cost, storage, and operational complexity.

---

## 1. Re-Evaluation of Rejected Items

### 3.2 Heartbeat — From REJECTED to CONDITIONALLY APPROVED

**Round 1 rationale for rejection:** The Heartbeat's scheduled scanning across Planner, SharePoint, and calendar would increase agent invocations by an estimated 22%, creating unpredictable cost spikes without proportional user value.

**What changes with owner's feedback:** The owner's point is structurally sound. If the single pane of glass is the product, then it is only as good as its coverage. A dashboard that shows emails and meetings but misses the Planner task due tomorrow or the document review that's been sitting for a week has a *completeness gap* — and completeness gaps erode trust. Users who learn the dashboard misses things stop checking it. The Heartbeat is not a luxury; it is the mechanism that closes the coverage gap for non-event-driven signals.

**What remains valid from Round 1:** Cost is not a theoretical concern. The current system invokes the triage agent per-email, per-Teams-message, and per-calendar-event. Adding Planner tasks, stale document reviews, and episodic memory pattern scans on a recurring schedule adds invocations that scale with time, not with user activity. A 15-minute heartbeat across 5 signal sources for 100 users is 48,000 additional agent invocations per day. That is real spend.

**Council compromise — the Scoped Heartbeat:**

Approve the Heartbeat with three constraints:

1. **Tiered frequency.** Not everything needs 15-minute polling. Calendar prep and Planner staleness run once daily (morning, tied to the existing Daily Briefing flow). SharePoint document review runs twice daily. Only genuinely time-sensitive sources (approaching deadlines, overdue tasks) get hourly checks. This reduces the invocation multiplier from 22% to an estimated 6-8%.

2. **Pre-triage in the flow, not the agent.** The existing v2.0 pattern already pre-filters calendar events with subject-matching (skip "Focus Time," "Lunch"). Extend this pattern to *all* Heartbeat sources. Planner tasks only invoke the agent if the due date is within 48 hours or the task was assigned in the last 24 hours. SharePoint documents only if shared within the last 72 hours and not yet opened. The agent should never see items the flow can trivially discard.

3. **Budget ceiling with telemetry.** Implement a daily invocation counter per user. If Heartbeat invocations exceed a configurable cap (default: 50/day), the Heartbeat degrades gracefully — it skips lower-priority sources rather than stopping entirely. Surface the counter in the Confidence Calibration dashboard so the admin can tune thresholds.

**Estimated cost impact with constraints:** 6-8% increase vs. 22% unscoped — within acceptable bounds for a POC, and the pre-triage filters will improve with sender intelligence data from Sprint 4.

### 3.3 Skill System — From BLOCKED to CONDITIONALLY APPROVED

**Round 1 rationale for blocking:** Open-ended user-defined prompts executing arbitrary queries create an unacceptable prompt injection surface. A user-crafted "Generate Weekly Report" skill could be socially engineered to exfiltrate data, escalate privileges, or produce misleading outputs.

**What changes with owner's feedback:** Two things, both significant. First, the Microsoft Orchestrator's built-in AI safety layers (content moderation, prompt injection detection, responsible AI filters) provide a platform-level sandbox that we were effectively ignoring. We do not need to build custom sandboxing because Copilot Studio *is* the sandbox. Second, the human-in-the-loop design means a skill's output is a *card on the dashboard*, not an executed action. A malicious skill that produces a misleading card is visible, reviewable, and dismissable — it cannot autonomously send emails, modify data, or escalate access. The blast radius of a compromised skill is a bad card, not a data breach.

**What remains valid from Round 1:** Platform safety layers are necessary but not sufficient. They protect against known prompt injection patterns but cannot guarantee defense against novel attacks. More importantly, the council's concern was never *only* about prompt injection. It was about **operational complexity**: a skill registry, versioning, debugging user-authored prompts, support burden when skills produce bad output, and the governance question of who is responsible when a skill creates cards that mislead a user's decision-making. These concerns are orthogonal to Microsoft's safety layers.

**Council compromise — the Governed Skill System:**

Approve the Skill System with four guardrails:

1. **Curated skill library, not open authoring.** V1 ships with 3-5 admin-published skills (Weekly Report, Project Status Summary, Pending Approvals Review, Stale Items Digest, Meeting Prep Bundle). Users can *enable/disable* skills but cannot *author* them. This eliminates the prompt injection surface from untrusted user input while delivering the extensibility value. User-authored skills can be a v2 feature with a separate governance review.

2. **Read-only data access.** Skills can query SharePoint lists, Planner boards, and Dataverse views — but they cannot write, update, or delete. Every skill output is a card with `card_status: PENDING`. The existing "prepare, never act" principle is the guardrail. Encode this as a hard constraint in the Skill Execution Flow: the only Dataverse operation a skill flow can perform is *Create* on `cr_assistantcard`.

3. **Skill output schema enforcement.** Every skill must produce output conforming to the existing `output-schema.json` contract. No new field types, no custom card formats, no arbitrary JSON. This means the PCF dashboard renders skill cards identically to triage cards — no new rendering surface, no new XSS vectors, no schema drift.

4. **Audit trail and kill switch.** Every skill invocation is logged with skill ID, user, timestamp, input parameters, and output card ID. An admin can disable any skill globally with immediate effect. Surface skill invocation counts in the existing Confidence Calibration dashboard.

## 2. Revised Priority Ranking (Single Pane of Glass Lens)

The ranking question becomes: *which enhancement brings the most missing work into the unified view?*

| Rank | Enhancement | Rationale |
|------|------------|-----------|
| **1** | **3.1 Episodic Memory** | Foundation for everything else. Without cross-session pattern recognition, the Heartbeat cannot identify staleness and skills cannot reference historical context. This is the hippocampus — build it first. |
| **2** | **3.2 Heartbeat (Scoped)** | Directly closes the completeness gap. The single pane of glass cannot be trusted if it only shows reactive signals. The scoped version manages cost while delivering coverage. |
| **3** | **3.4 Planner / Tasks Integration** | Highest-value missing signal source. Task assignments are explicit action requests that belong in the unified view. Implementation follows the proven flow pattern (trigger → compose → agent → Dataverse). Low architectural risk. |
| **4** | **3.3 Skill System (Governed)** | Extensibility multiplier. Once the Heartbeat and Planner populate the view with platform signals, skills let admins add *domain-specific* signals (project status, approval queues, compliance deadlines). Ranked below Planner because Planner is a known pattern; skills require new infrastructure. |
| **5** | **3.5 SharePoint Knowledge Connector** | Valuable for document-sharing signals but lower frequency than email/Teams/Planner. Many SharePoint notifications already generate email, which the existing EMAIL flow captures. Incremental value is moderate. |
| **6** | **3.6 Cross-Signal Threading** | Architecturally elegant but user-invisible. Conversation clustering (v2.0 Sprint 1B) already groups related cards. Deep cross-signal threading (linking a calendar prep card to an email thread to a Planner task) is a quality-of-life improvement, not a coverage expansion. Defer to v3.1. |

## 3. Implementation Plan Changes

**New artifacts required:**

- **Heartbeat Configuration Table** (`cr_heartbeatconfig`): Per-source frequency, pre-filter rules, daily invocation cap. One row per signal source per environment.
- **Heartbeat Execution Flow** (Flow 11): Scheduled recurrence with per-source branching, pre-triage filter logic, budget counter check, and degradation path.
- **Skill Registry Table** (`cr_skillregistry`): Skill ID, display name, prompt template, target data source, enabled/disabled flag, admin owner.
- **Skill Execution Flow** (Flow 12): Reads enabled skills for the current user, executes each skill's query, passes results through the triage agent with the skill's prompt template, writes output cards. Constrained to Create-only on `cr_assistantcard`.
- **Skill Audit Table** (`cr_skillauditlog`): Invocation log with skill ID, user, timestamp, card output ID, execution duration.

**Modified phases:**

- **Phase 20 (new): Episodic Memory** — 1 week. New `cr_episodicmemory` table, memory extraction flow triggered by card outcome changes, memory retrieval action registered with the triage agent. Builds on Sprint 1's outcome tracking.
- **Phase 21 (new): Scoped Heartbeat** — 1 week. Heartbeat config table, Flow 11 with tiered scheduling, pre-triage filters, budget ceiling. Depends on Phase 20 (Heartbeat uses episodic memory for staleness detection).
- **Phase 22 (new): Planner Integration** — 3 days. Standard flow pattern — trigger on Planner task assignment/update, compose payload, invoke agent. Follows the proven EMAIL/TEAMS/CALENDAR template.
- **Phase 23 (new): Governed Skill System** — 1.5 weeks. Skill registry table, execution flow, audit table, 3-5 seed skills, admin enable/disable UI in Canvas app. Highest complexity due to new infrastructure.

**Adjusted effort estimate:** 4-5 weeks for Phases 20-23, comparable to the v2.0 Second Brain sprint timeline. Phases 20-22 can be compressed with the established flow patterns. Phase 23 is the risk item.

## 4. Concerns That Remain Valid

The owner's mitigations address security and autonomy concerns. They do not address:

1. **Dataverse storage growth.** The Heartbeat creates cards for items that were previously invisible to the system. More visibility means more rows. The existing 30-day retention policy for EXPIRED/DISMISSED cards helps, but Heartbeat cards that are never acted on (because the user doesn't care about that Planner board) will accumulate. **Mitigation:** Add a "Heartbeat card auto-expire" threshold — if a Heartbeat source generates 10 consecutive DISMISSED/EXPIRED cards, reduce its scan frequency automatically.

2. **Agent capacity consumption at scale.** Even the scoped Heartbeat adds invocations. Skills add more. For a POC with 5-10 users this is fine. For enterprise rollout with 500+ users, capacity planning becomes critical. The POC-scope decision (documented in ROADMAP.md) defers this, which is acceptable — but it must be the *first* item addressed if the POC succeeds.

3. **Skill support burden.** Even with curated (admin-authored) skills, someone must maintain them. When the SharePoint list schema changes, the Weekly Report skill breaks. When Planner boards are reorganized, the Project Status skill produces empty cards. This is operational toil that compounds. **Mitigation:** Each skill should include a self-test query that runs before the main execution. If the self-test returns zero results or errors, the skill auto-disables and notifies the admin rather than producing empty cards.

4. **Cognitive overload risk.** The single pane of glass value proposition has a ceiling. More signals in the dashboard is better — until it isn't. If the Heartbeat and Skills push the daily card count from 30 to 120, the user's experience degrades even though coverage improved. **Mitigation:** The existing Daily Briefing Agent (v2.0 Sprint 2) becomes more important, not less. It must synthesize across all sources — including Heartbeat and Skill cards — so the user can start with "here are your 5 priorities" rather than scrolling 120 cards.

---

## Verdict

The council reverses its Round 1 rejection of the Heartbeat and lifts the block on the Skill System, subject to the constraints documented above. Both features are structurally necessary for the single pane of glass to deliver on its promise. The owner was right to push back — we over-indexed on risk and under-indexed on the product's core value proposition.

The constraints are not concessions to bureaucracy. They are engineering guardrails that make these features *shippable* rather than *aspirational*. A scoped Heartbeat with pre-triage filters and budget ceilings is a feature that can be monitored, tuned, and defended in a capacity review. An ungoverned Heartbeat polling everything every 15 minutes is a cost incident waiting to happen. A curated skill library with schema enforcement and audit logging is a feature an admin can support. An open-authoring skill system with arbitrary prompts is a governance nightmare regardless of platform safety layers.

Ship the constrained versions. Measure. Expand scope based on data.

**Council vote:** 4-0 APPROVE with constraints. No dissents. No abstentions.
